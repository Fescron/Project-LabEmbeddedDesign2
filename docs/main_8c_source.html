70  * ******************************************************************************
71  *
72  * @section SETTINGS Settings using definitions in dbprint and delay functionality
73  *
74  * In the file `debugging.h` dbprint functionality can be enabled/disabled with
75  * the definition `#define DEBUGGING`. If this line is commented, all dbprint
76  * statements are disabled throughout the source code because they're all
77  * surrounded with `#ifdef DEBUGGING ... #endif` checks.
78  *
79  * In the file `delay.h` one can **choose between SysTicks or RTC sleep functionality**
80  * for delays. This can be selected with the definition `#define SYSTICKDELAY`.
81  * If this line is commented, the RTC compare sleep functionality is used.
82  * Otherwise, delays are generated using SysTicks.
83  *
84  * In the file `delay.h` one can also **choose between the use of the ultra low-frequency
85  * RC oscillator (ULFRCO) or the low-frequency crystal oscillator (LFXO)** when being
86  * in a delay or sleeping. If the ULFRCO is selected, the MCU sleeps in EM3 and if
87  * the LFXO is selected the MCU sleeps in EM2. This can be selected with the definition
88  * `#define ULFRCO`. If this line is commented, the LFXO is used as the clock source.
89  * Otherwise, the ULFRCO is used.
90  *
91  * @note Check the next section for more info about this.
92  *
93  * ******************************************************************************
94  *
95  * @section CLOCKS1 Crystals and RC oscillators (delay.c)
96  *
97  * Normally using an external oscillator/crystal uses less energy than the internal
98  * one. This external oscillator/crystal can however be omitted if the design goal
99  * is to reduce the BOM count as much as possible.
100  *
101  * In the delay logic, it's possible to select the use of the ultra low-frequency
102  * RC oscillator (ULFRCO) or the low-frequency crystal oscillator (LFXO) when being
103  * in a delay or sleeping. If the ULFRCO is selected, the MCU sleeps in EM3 and if
104  * the LFXO is selected the MCU sleeps in EM2.
105  *
106  * @warning After testing it was noted that the ULFRCO uses less power than the
107  * LFXO but was less precise for the wake-up times. Take not of this when selecting
108  * the necessary logic!
109  *
110  * For the development of the code for the RN2483 shield from DRAMCO, it's possible
111  * they chose to use the LFXO instead of the ULFRCO in sleep because the crystal was
112  * more stable for high baudrate communication using the LEUART peripheral.
113  *
114  * @note For low-power development it's advised to consistently enable peripherals and
115  * oscillators/clocks when needed and disable them afterwards. For some methods it can
116  * be useful to provide a boolean argument so that in the case of sending more bytes
117  * the clock can be disabled only after sending the latest one. Perhaps this can be
118  * used in combination with static variables in the method so they keep their value
119  * between calls and recursion can be used.
120  *
121  * ******************************************************************************
122  *
123  * @section Initializations
124  *
125  * @warning Initializations for the methods `led(bool enabled);`, `delay(uint32_t msDelay);`
126  * and `sleep(uint32_t sSleep);` happen automatically. This is why their first call
127  * sometimes takes longer to finish than later ones.
128  *
129  * ******************************************************************************
130  *
131  * @section cmuClock_GPIO
132  *
133  * At one point in the development phase the clock to the GPIO peripheral was
134  * always enabled when necessary and disabled afterwards. Because the GPIO
135  * clock needs to be enabled for almost everything, even during EM2 so the MCU
136  * can react (and not only log) pin interrupts, this behavior was later scrapped.
137  *
138  * @todo Also talk about cmuClock_HFPER?
139  *
140  * ******************************************************************************
141  *
142  * @section RTCC RTCC (RTC calendar)
143  *
144  * At another point in the development phase there was looked into using
145  * the RTCC (RTC calendar) to wake up the MCU every hour. This peripheral can
146  * run down to EM4H when using LFRCO, LFXO or ULFRCO. Unfortunately the Happy
147  * Gecko doesn't have this functionality so it can't be implemented in this case.
148  *
149  * ******************************************************************************
150  *
151  * @section EM Energy modes (EM1 and EM3)
152  *
153  * At one point a method was developed to go in EM1 when waiting in a delay.
154  * This however didn't seem to work as intended and EM2 would also be fine.
155  * Because of this, development for this EM1 delay method was halted.
156  * EM1 is sometimes used when waiting on bits to be set.
157  *
158  * When the MCU is in EM1, the clock to the CPU is disabled. All peripherals,
159  * as well as RAM and flash, are available.
160  *
161  * In EM3, high and low frequency clocks are disabled. No oscillator (except
162  * the ULFRCO) is running. Furthermore, all unwanted oscillators are disabled
163  * in EM3. This means that nothing needs to be manually disabled before
164  * the statement `EMU_EnterEM3(true);`.
165 
166  * The following modules/functions are are generally still available in EM3:
167  * @li I2C address check
168  * @li Watchdog
169  * @li Asynchronous pin interrupt
170  * @li Analog comparator (ACMP)
171  * @li Voltage comparator (VCMP)
172  *
